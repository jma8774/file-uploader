# TICKET-018: GET /d/:token

## Status

Backlog

## Goal

Stream the actual file bytes for an active token, with the original filename in `Content-Disposition`, and atomically bump the download counters.

## Background

This is the only endpoint the browser hits directly via `<a href download>`. Everything else is JSON. Heavy abuse-protection (per-IP rate limits, the global safety cap) lands in TICKET-021; here we cover the happy path plus the per-file caps that are inseparable from "should we serve this byte stream at all."

## Requirements

- `backend/src/routes/download.ts` registers `GET /d/:token`.
- Lookup the file by token (reusing `fileService.getActiveByToken` where possible). If not found / deleted / expired → 404 with `{ error: 'EXPIRED', message: 'This download link is no longer available.' }`.
- If `download_count >= max_downloads` → 403 with `{ error: 'MAX_DOWNLOADS_REACHED', message: 'This file has reached its download limit.' }`.
- If `bytes_downloaded_estimate + size_bytes > max_transfer_bytes` (per-file 2 GiB cap from the spec) → 403 with `{ error: 'MAX_DOWNLOADS_REACHED', message: 'This file has reached its transfer limit.' }`. (Same code; same user message family.)
- Increment counters **before** streaming (so a client disconnect mid-stream doesn't drop the count). All increments happen inside one transaction:
  - `UPDATE files SET download_count = download_count + 1, bytes_downloaded_estimate = bytes_downloaded_estimate + size_bytes WHERE id = ?`
  - `INSERT INTO download_events (file_id, downloaded_at, bytes, ip_hash, user_agent) VALUES (?, ?, ?, ?, ?)`
  - `INSERT INTO bandwidth_events (file_id, bytes, event_type, created_at) VALUES (?, ?, 'download', ?)`
- IP hashing: HMAC-SHA256 of the request IP using `config.ipHashSecret`. Store the hex digest. Truncate the user agent string to 256 chars before insert.
- Serve the file with `res.download(absolutePath, originalName)` — Express handles the headers. The `absolutePath` is `path.join(config.uploadDir, file.stored_name)`.
- Resolve the absolute path safely: after `path.resolve`, verify it starts with the resolved `config.uploadDir` to defend against any future bug that lets a `stored_name` traverse out of the uploads dir. If it doesn't, log a critical error and 500. (Belt + suspenders — `stored_name` is generated by us, but the check costs nothing.)

## Acceptance criteria

- After TICKET-016 uploads a file at `/tmp/sample.txt`, `curl -OJ http://localhost:3000/d/<token>` downloads it back with the original filename.
- The `files` row's `download_count` and `bytes_downloaded_estimate` increment by exactly 1 / size on each successful download.
- A `download_events` row and a `bandwidth_events` row are inserted with each successful download.
- Hitting the URL with a token that's not in the DB → 404 + spec error JSON.
- After download_count reaches max_downloads, the next request → 403 + spec error JSON.
- `Content-Disposition` carries the original filename even when the on-disk file is `<random>.bin`.

## Out of scope

- Per-IP rate limiting / hourly caps — TICKET-021.
- Global safety cap enforcement — TICKET-021.
- `X-Accel-Redirect` for offloading to Nginx — explicit future optimization per spec §9.
- Resume / range requests — not required in v1.

## Notes for implementation

- The order matters: validate, then increment counters, then start the stream. Don't trust the spec-recommended order of "increment after send" — that's lossier on flaky networks.
- `res.download` already sets `Content-Disposition: attachment; filename="..."` with appropriate encoding for non-ASCII names.
- The path-traversal check is the kind of defensive code that's actually load-bearing; keep it in.
