# Glossary

## Token

A random, URL-safe string (16 random bytes → base64url) that identifies a file in `/file/:token` and `/d/:token`. Treated as a secret-ish handle: anyone with the token can download. Generated by the backend using secure randomness.

## Download page

The page at `/file/:token`. Shows filename, size, expiration, and a download button. Does **not** itself trigger the download — clicking the button hits `/d/:token`.

## Direct download URL

`/d/:token`. The endpoint that actually streams the file bytes. Increments download/bandwidth counters server-side.

## Expiration

Each file has a TTL (24 h in v1). After expiration the file is deleted from disk and marked `is_deleted` in the DB. Subsequent requests show the "expired" state.

## Max downloads

Per-file cap computed from `2 GiB / fileSize` (floor, min 1). Protects against a single file consuming the monthly transfer budget.

## Per-file transfer cap

The 2 GiB ceiling that `maxDownloads` is derived from.

## Monthly transfer safety cap

Global 250 GiB ceiling on estimated monthly outbound transfer. When hit, uploads and downloads are disabled and the homepage shows a "safety limit reached" message.

## Storage cap

6 GiB local disk ceiling. Cleanup job deletes oldest active files first to stay under this.

## Cleanup job

Backend job (every ~10 min) that deletes expired files and trims storage if over the cap.

## IP hash

A salted hash of the requester's IP, stored on download events for rate limiting and rough abuse detection. Never logged in plaintext, never shown publicly.

## Active file

A file in the DB with `is_deleted = 0` and `expires_at` in the future. Counted in the homepage stats.

## Upload state

Frontend state machine: `idle | selected | uploading | success | error`.

## Emulated / EMULATED

Frontend-first marker for code paths that fake a backend response. Search for `// EMULATED:` to find them when wiring up the real backend.
